# Восстановление пароля (Password Reset) — дизайн

**Дата:** 2026-06-16
**Статус:** утверждён, готов к написанию плана реализации

## Контекст и решение

Пользователи MEDIZO иногда забывают пароль и не могут войти. Нужен дешёвый и
простой механизм восстановления.

**Важно:** «прислать текущий пароль на почту» технически невозможно — пароли
хранятся как односторонний bcrypt-хеш (`User.passwordHash`, `src/lib/auth.ts:28`,
`src/app/actions.ts:122`). Исходный пароль из хеша не восстанавливается, и хранить/
пересылать пароли открытым текстом — грубая уязвимость. Поэтому выбран стандартный
флоу сброса по одноразовой ссылке.

### Принятые решения

- **Механизм:** ссылка для сброса (reset link), не 6-значный код.
- **Отправка писем:** Resend (3000 писем/мес бесплатно, простой API, хорошая
  доставляемость).
- **Кто может сбрасывать:** только одобренные пользователи (`approvedAt != null`).
- **Объём:** только флоу «Забыли пароль?». Смену пароля в профиле для залогиненных
  не делаем.

## Поток

```
Страница входа
   └─ ссылка «Забыли пароль?»  →  /forgot-password
                                      │  email → requestPasswordReset(email)
                                      │  всегда: «Если такой email есть — письмо отправлено»
                                      ▼
                            Письмо (Resend) со ссылкой
                            {NEXTAUTH_URL}/reset-password?token=XXXX
                                      │  клик
                                      ▼
                                /reset-password
                                      │  новый пароль + повтор → resetPassword({token, newPassword})
                                      ▼
                              /login  (тост «Пароль изменён»)
```

## Архитектура

### Модель данных

Новая таблица `PasswordResetToken` (отдельная от `User`):

```prisma
model PasswordResetToken {
  id        String    @id @default(cuid())
  userId    String
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  tokenHash String    @unique   // SHA-256 хеш токена, НЕ сам токен
  expiresAt DateTime
  usedAt    DateTime?
  createdAt DateTime  @default(now())
  @@index([userId])
}
```

В модель `User` добавляется обратная связь: `passwordResetTokens PasswordResetToken[]`.
Нужна Prisma-миграция (`npm run db:migrate`).

**Ключевой инвариант безопасности:** в БД лежит SHA-256-**хеш** токена, а сам
сырой токен попадает только в письмо. Утечка БД не даёт сбросить пароль. Токен
высокоэнтропийный (`crypto.randomBytes(32)`), поэтому быстрый SHA-256 достаточен —
bcrypt здесь не нужен.

### Отправка писем — `src/lib/email.ts`

Новый модуль с функцией `sendEmail({ to, subject, html })` поверх пакета `resend`.

- Env-переменные: `RESEND_API_KEY`, `EMAIL_FROM` (например `MEDIZO <noreply@docjob.kz>`).
- Базовый URL ссылки берётся из существующей `NEXTAUTH_URL`.
- **Fallback для локальной разработки:** если `RESEND_API_KEY` не задан, `sendEmail`
  печатает письмо/ссылку в консоль вместо реальной отправки. Это позволяет
  прогонять весь флоу без ключей и без верифицированного домена.
- Шаблон письма сброса (HTML) — простой, с одной кнопкой-ссылкой и упоминанием
  срока действия (1 час).

### Серверные экшены — `src/app/actions.ts`

Обе функции возвращают `ActionResult<T>` в существующем стиле проекта.

**`requestPasswordReset(email: string)`**
1. Находим юзера по `email.toLowerCase()`.
2. **Ответ всегда одинаковый успех**, независимо от того, есть юзер или нет
   (anti-enumeration — нельзя перебором узнать, кто зарегистрирован).
3. Если юзер существует **и** `approvedAt != null`:
   - генерим сырой токен `crypto.randomBytes(32).toString('hex')`;
   - вычисляем SHA-256-хеш, гасим прошлые неиспользованные токены этого юзера,
     сохраняем новую строку с `expiresAt = now + 1ч`;
   - вызываем `sendEmail` со ссылкой `{NEXTAUTH_URL}/reset-password?token={raw}`.
4. Анти-флуд: если активный (неиспользованный, не просроченный) токен создан
   менее 60 секунд назад — новое письмо не шлём.

**`resetPassword({ token: string, newPassword: string })`**
1. Хешируем входящий токен (SHA-256), ищем строку по `tokenHash`.
2. Проверяем: существует, `usedAt == null`, `expiresAt > now`.
3. Валидируем `newPassword` теми же правилами, что при регистрации (мин. длина).
4. `bcrypt.hash(newPassword, 10)` → обновляем `user.passwordHash`.
5. Ставим `usedAt = now`, гасим остальные токены этого юзера.

(Опционально — лёгкая проверка токена для страницы, чтобы сразу показать
«ссылка устарела» до ввода пароля. Можно сделать отдельным легковесным экшеном
или переиспользовать логику проверки.)

### Страницы и UI

- **`/forgot-password`** — поле email + кнопка. После отправки — нейтральное
  сообщение «Если email зарегистрирован, мы отправили ссылку».
- **`/reset-password`** — читает `?token=` из URL, поля «новый пароль» + «повтор».
  На входе лёгкая проверка токена → при невалидном токене показываем
  «Ссылка устарела или недействительна» и ссылку на повторный запрос.
- **Страница входа** (`src/app/login/page.tsx`) — добавить ссылку «Забыли пароль?».
- Обе новые страницы **публичные** → добавить их пути в список public paths в
  `src/middleware.ts`, иначе middleware редиректнёт незалогиненного на `/login`.

## Безопасность

- Токен одноразовый, срок жизни 1 час, в БД хранится SHA-256-хешем.
- Одинаковый ответ для существующих и несуществующих email (anti-enumeration).
- Сброс доступен только одобренным пользователям (`approvedAt != null`).
- Новый пароль валидируется (правила как при регистрации).
- Анти-флуд: не чаще одного письма в 60 секунд на пользователя.

## Обработка ошибок

- `requestPasswordReset`: ошибки отправки письма логируются на сервере, но
  пользователю всё равно показывается нейтральный успех (не палим, существует ли
  email и работает ли почта).
- `resetPassword`: явные ошибки — «ссылка устарела/недействительна», «пароль
  слишком короткий», «пароли не совпадают» (последнее — на клиенте).

## Тестирование

- Юнит/интеграционно: генерация и одноразовость токена; просроченный токен
  отклоняется; использованный токен отклоняется; неодобренный юзер не получает
  письма; одинаковый ответ для существующего и несуществующего email; новый
  пароль действительно меняет `passwordHash` и проходит вход.
- Ручной прогон в dev через консольный fallback `sendEmail` (без ключей Resend).

## Что НЕ делаем (YAGNI)

Смена пароля в профиле, SMS, magic-link вход, rate-limiting на Redis,
кастомный HTML-дизайнер писем.

## Внешние шаги (не код)

Завести аккаунт Resend, получить `RESEND_API_KEY`, добавить DNS-записи (SPF/DKIM)
для `docjob.kz`. Для быстрого теста Resend разрешает слать с `onboarding@resend.dev`
на собственную верифицированную почту без верификации домена.
